# Agent 开发审视 Skill

## 使用时机

当以下情况发生时使用此 skill：

1. **用户提出新功能需求**
   - "我要实现一个权限系统"
   - "添加重试机制"
   - "需要缓存功能"

2. **发现未集成模块**
   - 代码存在但未被使用
   - 导出但无引用
   - 测试目录为空

3. **架构决策前**
   - "是否需要这个模块？"
   - "这个功能是必需的吗？"
   - "应该保留还是删除？"

4. **代码审查时**
   - 发现大量未使用代码
   - 模块之间缺乏关联

## 审视流程

### 第一步：理解需求

**询问用户以下问题**（根据实际情况选择）：

1. **功能目的**
   - 这个功能要解决什么问题？
   - 有什么具体的使用场景？
   - 能否举一个实际例子？

2. **集成计划**
   - 计划在哪里使用这个功能？
   - 哪个模块会调用它？
   - 有没有集成的时间表？

3. **替代方案**
   - 现有功能能否满足需求？
   - 有没有更简单的实现方式？
   - 其他框架是怎么做的？

### 第二步：检查现有实现

**执行以下检查**：

1. **代码存在性检查**

   ```bash
   # 查找相关文件
   find src -name "*<功能名>*" -type f
   find tests -name "*<功能名>*" -type f

   # 检查导出
   grep -r "export.*<功能名>" src/index.ts

   # 检查使用
   grep -r "from.*<功能名>\|import.*<功能名>" src/
   ```

2. **使用情况分析**
   - 哪些文件导入了这个模块？
   - 实际调用了哪些方法？
   - 示例代码中是否使用？

3. **测试覆盖检查**
   - 是否有测试文件？
   - 测试是否通过？
   - 测试覆盖率如何？

### 第三步：必要性评估

**基于以下维度评分**（1-5分）：

| 维度           | 评分标准               |
| -------------- | ---------------------- |
| **需求明确性** | 1=模糊，5=非常明确     |
| **使用频率**   | 1=几乎不用，5=高频使用 |
| **集成必要性** | 1=可有可无，5=不可或缺 |
| **实现成本**   | 1=成本极高，5=成本很低 |
| **维护负担**   | 1=负担重，5=负担轻     |

**总分计算**：需求明确性 × 0.3 + 使用频率 × 0.3 + 集成必要性 × 0.4

**决策矩阵**：

| 总分范围 | 建议                    |
| -------- | ----------------------- |
| 4.0-5.0  | ✅ 立即实现并集成       |
| 3.0-3.9  | 🟡 计划实现，设定时间表 |
| 2.0-2.9  | 🟠 推迟，等待明确需求   |
| 1.0-1.9  | 🔴 删除或废弃           |
| 0        | ❌ 不实现               |

### 第四步：集成点分析

**如果决定保留，分析集成点**：

1. **核心模块集成**
   - agent.ts 是否需要调用？
   - factory.ts 是否需要支持？
   - 其他核心模块是否依赖？

2. **配置管理**
   - 是否需要配置项？
   - 配置项放在哪里？
   - 是否需要验证？

3. **错误处理**
   - 失败时如何处理？
   - 是否需要降级？
   - 是否需要重试？

4. **日志和监控**
   - 需要记录哪些日志？
   - 是否需要监控指标？
   - 如何追踪使用情况？

```
## 审视原则

### 避免过度设计

**警惕信号**：
- "其他框架有，我们也要有"
- "为了完整性"
- "以后可能会用到"

**正确思路**：
- 先分析实际需求
- 再考虑实现方案
- 最后评估必要性

### 遵循铁律

**铁律第一条**：不要着急实现，先分析必要性

**执行检查清单**：
- [ ] 是否有明确的使用场景？
- [ ] 是否有具体的使用者？
- [ ] 是否有集成的时间表？
- [ ] 是否有测试验证？
- [ ] 是否有文档说明？

**如果任何一个答案是否定的，暂缓实现。**

### 死代码识别

**死代码特征**：
1. 导出但未被任何模块引用
2. 有类型定义但无实现
3. 有测试目录但为空
4. 示例代码中未使用
5. 文档中未提及

**处理建议**：
- 删除死代码，减少维护负担
- 或者补充集成逻辑，让功能真正可用

## 案例参考

### 案例1：权限系统（❌ 反面教材）

**问题**：
- 实现了完整的 PermissionSystem（162行）
- agent.ts 完全未使用
- 测试目录为空
- 示例代码未使用

**根本原因**：
- 看到其他框架有，就也实现一个
- 没有分析实际需求
- 没有计划集成时间表

**正确做法**：
1. 先问：谁需要权限系统？
2. 再问：在什么场景下使用？
3. 最后：什么时候集成？

### 案例2：插件系统（✅ 正面案例）

**成功因素**：
- agent.ts 集成使用
- 示例代码实际使用（inferhub-plugin）
- 有明确的扩展点

**关键点**：
- 有实际使用场景
- 有集成代码
- 有示例验证

## 输出格式

**最终输出应包含**：

1. **简洁的决策建议**（1句话）
2. **详细的理由说明**（3-5句话）
3. **具体的行动清单**（可执行的任务）
4. **风险评估**（潜在问题和缓解措施）

**示例**：

```

决策建议：🔴 删除权限系统模块

理由：该模块已实现162行代码，但未被任何核心模块使用，
测试目录为空，示例代码未使用。属于典型的"为了实现而实现"，
违反了铁律"先分析必要性"。

行动清单：

- 删除 src/permissions/ 目录
- 从 src/index.ts 移除导出
- 删除 tests/permissions/ 空目录
- 更新相关文档

风险：无风险，纯清理工作。

```

## 注意事项

1. **不要害怕删除代码**
   - 死代码是技术债务
   - 删除比保留更安全
   - Git 历史可以找回

2. **优先考虑集成**
   - 未集成的代码毫无价值
   - 集成比实现更重要
   - 示例是最好的验证

3. **保持简单**
   - 复杂度会带来维护负担
   - 简单的设计更容易理解
   - 避免过度抽象

4. **重视反馈**
   - 用户反馈是最真实的
   - 实际使用场景最重要
   - 数据胜于直觉
```
