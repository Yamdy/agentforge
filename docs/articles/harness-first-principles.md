# Harness：让 AI 从 demo 到生产的那层东西

你写了一个 agent。循环调 LLM，解析 tool call，执行工具，返回结果。跑了两三个 example，效果不错。你觉得可以上线了。

然后真正的任务来了。跑了 20 多步，中间不知道什么时候开始偏了。你在终端前看着它反复调同一个接口，参数每次略有不同，但没有一次是对的。你想让它停下来，但发现你没有任何办法在它运行时中断它——除了 kill 进程，而 kill 进程意味着之前的全部工作白费。

这不是一个假想的场景。这是每一个把 agent 从 demo 推向生产的人都会遇到的事。

## 所以 harness 到底是什么

2026 年初 LangChain 提了一个公式，Agent = Model + Harness。被引用了很多次，但说实话定义一直很模糊。Harness 是 "model 之外的一切"？那一条 system prompt 算不算？一个 tool schema 算不算？边界在哪。

我更愿意用一个问题来定义它：你的 demo 在生产环境会怎么挂？

Harness 就是针对每一种挂法的对策。它不是一组固定的模块，而是一系列故障驱动的机制。出了什么问题，就加对应的东西。没出问题，就不需要。

这听起来像是废话，但实际上它给出了一个很实用的判断标准：harness 里的每一个组件，都编码了一个 "模型自己做不了" 的假设。step budget 存在，是因为模型可能无限循环。kill switch 存在，是因为你可能在运行时需要强制中断。checkpoint 存在，是因为 session 断了就没了。如果有一天模型能自己判断什么时候停、停下来之后能从断点恢复，那 kill switch 和 checkpoint 就可以去掉。

换句话说，模型越强，harness 越薄。但不会消失。后文会解释为什么。

## 那到底会怎么挂

先看一眼 agent 循环。一个没有 harness 的循环就是左边的样子——LLM 决定做什么就做什么，没人检查、没人拦、没人记录。加上 harness 之后，每一步都有对应的机制在起作用：

```
  没有 harness                          有 harness
  ────────────                          ──────────

  ┌───────────┐                        ┌───────────┐
  │  组装上下文 │                        │  组装上下文  │
  └─────┬─────┘                        └─────┬─────┘
        │                                    │
        │                              ① 注入原始目标 + 进度  ← 防走偏
        │                              ② 检查步数/费用预算    ← 防失控
        │                                    │
        ▼                                    ▼
  ┌───────────┐                        ┌───────────┐
  │  调 LLM   │                        │   调 LLM   │
  └─────┬─────┘                        └─────┬─────┘
        │                                    │
        │                                    │
        ▼                                    ▼
  ┌───────────┐                        ┌───────────┐
  │  执行工具  │                        │  ③ 校验工具调用  │ ← 防越权
  └─────┬─────┘                        │  ④ 沙箱隔离执行  │
        │                              └─────┬─────┘
        │                                    │
        │                              ⑤ 记录 trace       ← 防静默失败
        │                              ⑥ 保存 checkpoint   ← 防失忆
        │                                    │
        ▼                                    ▼
      继续？                              ⑦ 验证输出         ← 防幻觉
      没人管                              ⑧ 积累经验         ← 防重犯
                                            │
                                            ▼
                                        压缩上下文           ← 防溢出
                                            │
                                            ▼
                                          继续？
                                      有人管，有预算，有记录
```

拆开来看，故障模式是有限的。我整理了八种，但说实话你不需要记住八种。你只需要知道，每一个都对应一个真实的生产事故。

**agent 跑着跑着忘了自己在干什么。** 这是最常见的。LLM 没有跨步骤的记忆，跑了十几步 tool call 之后，最初的指令已经稀释得差不多了。有人实测过 5 步的 agent 传递链，原始目标从 47 个词被压缩到 12 个词，关键约束丢失。解法不复杂：每一步自动把原始目标和当前进度注入上下文。不是让模型 "记住"，是让它 "被提醒"。Anthropic 用 progress file 做这件事，有些系统做得更自动化——每轮循环注入一个结构化的状态块，agent 不需要主动去读，就像呼吸一样。

**agent 停不下来。** 遇到模糊指令或工具报错，模型可能反复重试。这在 demo 里不是问题，在生产里就是真金白银。解法是硬上限：步数、费用、时间，任一触达就强制终止。这不是建议，是系统级约束，模型绕不过去。但光有上限还不够。58% 的企业说自己监控 agent 运行状态，但只有不到 40% 说自己能实际停掉一个失控的 agent。监控不等于控制。

**session 断了，白干了。** context window 是有限的，长任务的对话历史可能几十万 token。session 结束或溢出，没做持久化就从零开始。checkpoint 是基本功。更进一步的做法是用 git 记录每个步骤的产物，配合文字摘要做交接。下一个 session 起来，先读摘要了解上下文，再查 diff 了解变更，然后接着干。本质上跟人类换班没区别。

**agent 一本正经胡说八道。** 这是 LLM 的固有特性。demo 里只是尴尬，生产里可能是灾难——尤其当 agent 有工具执行权限的时候。解法是验证：工具结果做 schema 校验，最终输出做质量检查。另一个模型做 cross-check 也行，人工审核也行。

**agent 做了不该做的事。** demo 里 agent 只有几个无害工具。生产里它可能连着数据库和支付接口。模型决定调一个删除操作，你不想让它执行的时候它已经执行了。allowlist 是第一道，hook 是第二道，沙箱是第三道。高风险操作走人工审批。不是可选项。

**agent 悄悄地错了，没人知道。** 不是崩溃才有问题。一个看似正常的输出可能实际是错的，如果没有任何 trace，事后 debug 基本不可能。每一步记录输入、输出、工具调用、token 消耗。结构化的 trace，不是散落的日志。

**同样的错反复犯。** 大多数 agent 是无状态的，不从错误中学习。有团队的做法是每次任务后反思，把经验蒸馏成 skill 文件，下次自动加载。不改模型权重，任何模型都能用。还有人提出了 harness 自演化——worker 执行、evaluator 打分、evolver 改 harness，让 harness 自己优化自己。

**对话太长，context 装不下了。** 截断会丢信息。渐进压缩是更常见的策略——从轻量的截断工具输出到重量的摘要对话历史。极端情况下直接做 context reset，从结构化交接文件重建一个新 session。

## 等一下，这些我好像在哪见过

把 agent 循环和控制回路放在一起看：

```
  经典控制回路                          Agent 循环
  ────────────                          ──────────

  ┌─────────┐                          ┌──────────┐
  │  传感器   │                         │  读取上下文 │
  │ (温度/压力)│                         │ + trace   │
  └────┬────┘                          └─────┬────┘
       │                                     │
       ▼                                     ▼
  ┌─────────┐                          ┌──────────┐
  │  控制器   │                         │   LLM    │
  │  (PID)   │                          │  (决策)   │
  └────┬────┘                          └─────┬────┘
       │                                     │
       ▼                                     ▼
  ┌─────────┐                          ┌──────────┐
  │  执行器   │                         │  工具调用  │
  │ (阀门/电机)│                         │ (API/DB) │
  └────┬────┘                          └─────┬────┘
       │                                     │
       ▼                                     ▼
  负反馈: 检测偏差                      事实注入: 检测目标偏移
  前馈: 预判拦截                        权限/沙箱: 预判拦截
  断路器: 防发散                        step budget: 防发散
```

如果你写过微服务，读到这里应该觉得眼熟。

step budget 和 kill switch，不就是断路器吗。工具调用的 allowlist 和参数校验，不就是 API gateway 做的事吗。全链路 trace，不就是分布式追踪吗。checkpoint 和断点恢复，不就是持久化队列吗。渐进压缩，不就是日志的 retention policy 吗。

事实注入——每一步自动把原始目标注入上下文——跟 PID 控制器做的事情在结构上是一样的。PID 检测偏差，注入修正信号。agent harness 检测目标偏移，注入原始指令。被控对象从温度变成了 LLM 的注意力，但控制回路没变。

这不是巧合。harness 本质上就是控制论在一个新被控对象上的应用。这个被控对象碰巧是概率性的而不是确定性的，但控制结构不变。级联防护、爆炸半径控制、优雅降级、混沌工程——微服务时代验证过的那些实践，换个名字就能用。

所以如果你在纠结 "harness 应该包含什么"，换个问法可能更有效：你的系统在生产环境会出现什么故障？然后针对每个故障加最小的机制。自下而上，不是自上而下。

## 另一个角度

写这篇文章的时候，我一直觉得 harness 和什么东西很像，但说不上来。后来想明白了——道德经第十一章："埏埴以为器，当其无，有器之用。" 揉泥巴做器皿，有用的不是泥巴，是中间的空。

结构上确实对得上。Model 是泥巴，是实体，是能力。Harness 不提供能力，它提供形状和空间。它定义边界，让能力在正确的范围内释放。没有形状的泥巴还是泥巴，但成不了器。

有一点特别贴切："模型越强，harness 越薄"——泥巴质量越好，对器皿形状的约束可以越少。但 "空" 永远需要。不管泥巴多好，没有器皿的形状，它还是一滩泥。

## 先做什么

故障的严重程度不同，优先级自然不同。

```
        ┌─────────────────────┐
        │    P2  防幻觉        │
        │    防重犯            │     锦上添花
        │    验证 loop         │     从好用 → 卓越
        ├─────────────────────┤
        │    P1  防走偏        │
        │    防溢出            │     长稳运行的前提
        │    防静默失败         │     跑几百步不崩
        ├─────────────────────┤
        │    P0  防失控        │
        │    防越权            │     没有这些不敢上线
        │    防失忆            │     基本要求
        └─────────────────────┘
```

最底层的三件事——防失控、防越权、防失忆——没有这些，你不敢让 agent 在生产环境跑。这不是工程选择，是基本要求。

再上一层——防走偏、防溢出、防静默失败——是 agent 要长稳运行的前提。跑几百步、跨 session、处理真实任务，没有事实注入和上下文压缩，跑得越久越不可靠。

最上面——防幻觉、防重犯——是锦上添花。验证 loop 和经验沉淀会让产品从能用变成好用，但不是上线的前提。

不要一口气全做。先过第一关，让产品能上线。再补第二关，让它能长跑。最后追第三关。

至于 harness 的终局，我其实不确定它有一个固定的终局形态。模型在变强，harness 在变薄。有些今天必须由 harness 做的事，明天模型自己就能做了。但 harness 不会消失——因为不管模型多强，你始终需要那个 "空"：边界、方向、起点、轨迹。这些不是智能，但它们让智能变得可靠。

先把第一关过了。
