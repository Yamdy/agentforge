# 设计理念

> 本文档定义 AgentForge 事件流架构的核心设计理念、思想起源和架构铁律。

---

## 从需求出发

| 需求 | 传统方案 | RxJS 方案 |
|------|---------|----------|
| **可观测** | 日志系统、Hook | `subscribe()` 天然可观测 |
| **可中断** | AbortController | `takeUntil()`, `unsubscribe()` |
| **可恢复** | Checkpoint 系统 | 状态快照 + 重建流 |
| **重试** | 重试中间件 | `retry()` 操作符 |
| **超时** | 超时中间件 | `timeout()` 操作符 |
| **打点** | Tracing Hook | `tap()` 操作符 |
| **HITL** | Permission Hook | `Subject` 暂停 + resume |

## Agent Harness 架构定位

> **Agent = LLM（认知决策核心）+ Harness（工程管控基座）**

AgentForge 定位为 **Agent 开发框架底座（Harness）**，核心理念是：

- **模型是智能核心**：LLM 负责推理、决策、语义理解
- **框架是工程基座**：AgentForge 负责执行管控、资源约束、状态持久、安全隔离、行为可观测
- **所有 Agent 行为必须经过 Harness 管控，不可绕过**

### Harness 六大核心模块（E/T/C/S/L/V）

| 模块 | 职责 | AgentForge 实现 |
|------|------|----------------|
| **E - 执行循环** | Agent 运行心跳 | `expand()` 递归驱动事件流 |
| **T - 工具注册表** | 外部能力统一网关 | `ToolRegistry` + Zod Schema 校验 |
| **C - 上下文管理器** | 对抗上下文腐烂 | `ContextBuilder` + 依赖注入 |
| **S - 状态存储** | 长任务可持续底座 | `Checkpoint` + 状态快照 |
| **L - 生命周期钩子** | 硬约束治理层 | `Plugin` 系统 + 拦截器/观察器 |
| **V - 评估接口** | 客观可观测校验层 | 事件流全链路可观测 |

## 为什么是 RxJS + Zod

| 选择 | 理由 |
|------|------|
| **RxJS** | 事件流天然可观测、可组合、可中断；操作符覆盖所有控制流需求 |
| **Zod** | 运行时类型验证 + TypeScript 类型推断；事件结构有保障 |
| **不用 Effect-TS** | 学习曲线更平缓；RxJS 生态成熟；Observable 模型直观 |
| **不用多套机制** | 单一模型降低心智负担；所有扩展用同一套操作符 |

## 落地铁律

> ⚠️ **重要**：以下铁律为架构设计的第一性约束，违反将导致系统不稳定或不可维护。

### 架构层（技术负责人必背）

| # | 铁律 | 说明 |
|---|------|------|
| 1 | **RxJS 管好订阅与销毁，杜绝内存泄漏** | 所有流必须 `takeUntil(destroy$)`，嵌套流继承父级销毁信号 |
| 2 | **Zod 统一全链路数据契约，防脏数据、防协议崩坏** | 外部数据 Tier 1 强校验 + 兜底降级；跨模块数据 Tier 2 Schema 契约 |
| 3 | **轻量 DI 解耦所有外部能力，方便替换与扩展** | 核心 Loop 只依赖接口；禁止内部 `new` 硬编码 |
| 4 | **Hook 插件异常隔离，横向能力零侵入** | 单插件报错不拖垮主循环；Hook 横向切片，DI 纵向替换 |
| 5 | **分层设计、作用域隔离** | 全局单例共享，会话状态独立；SubAgent 进程内，A2A 跨进程 |
| 6 | **流程串行严格管控，副作用异步并行** | 拦截器 `concatMap` 阻塞主流程；观察器 `tap` 不阻塞，异常静默 |

### 实现层（开发者必背）

| # | 铁律 | 说明 |
|---|------|------|
| 1 | **严格隔离同步事务代码，不在 RxJS 流里写强同步锁** | RxJS 流是异步的，同步锁会破坏流语义；用 `concatMap` 保证串行 |
| 2 | **全局统一 `destroy$ + takeUntil`，禁止裸订阅** | 任何 `subscribe()` 必须配 `takeUntil(destroy$)`，否则内存泄漏 |
| 3 | **插件钩子强制独立 try/catch，禁止击穿主流程** | 每个插件包裹 `catchError`，错误降级处理，绝不 throw 到主循环 |
| 4 | **核心主流程 `concatMap` 串行，副作用 `mergeMap` 并行，杜绝乱并发** | 拦截器用 `concatMap`，观察器用 `tap`/`mergeMap`，不可混用 |
| 5 | **内部可信数据简化 Zod，外部 LLM/三方输入强校验** | Tier 3 内部仅 TypeScript 类型；Tier 1 外部 Zod 强校验 + 兜底 |
| 6 | **封装统一流调试工具：链路ID、阶段打点、流状态快照** | 降低 RxJS 调试难度；每个事件带 `sessionId`/`correlationId` |

## 铁律约束矩阵

```
┌─────────────────────────────────────────────────────────────────┐
│                    铁律违反后果                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  架构层违反 → 系统级故障                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ #1 内存泄漏 → 服务崩溃                                    │   │
│  │ #2 数据脏 → LLM 幻觉、工具调用失败                        │   │
│  │ #3 耦合紧 → 无法替换 LLM/Memory，僵化                     │   │
│  │ #4 插件击穿 → 主循环崩溃，Agent 不可用                    │   │
│  │ #5 作用域混乱 → 多 Agent 上下文串数据                     │   │
│  │ #6 并发混乱 → 竞态条件，结果不可预测                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
│  实现层违反 → 代码级 Bug                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ #1 同步锁 → 死锁、流阻塞                                  │   │
│  │ #2 裸订阅 → 订阅泄漏，内存增长                            │   │
│  │ #3 异常穿透 → 单插件拖垮整个 Agent                        │   │
│  │ #4 操作符混用 → 状态竞态，上下文错乱                      │   │
│  │ #5 过度校验 → 性能下降，循环内每步校验                    │   │
│  │ #6 缺调试工具 → RxJS 流难以排查                           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```
