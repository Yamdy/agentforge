# Harness 概念总结

基于六篇文章及深度讨论。

---

## 一、核心定义

> **Agent = Model + Harness**

Harness 是**包裹在 AI 模型外面的所有代码、配置和执行逻辑**——模型提供原始推理能力，Harness 提供一切让这种能力变得可靠、安全、可用的工程基础设施。

最直观的类比：

| 类比 | 含义 |
|------|------|
| **马与马具** | 模型是马（强大但不可预测），Harness 是缰绳、马鞍、嚼子——将力量引导到可控方向 |
| **操作系统** | Model = CPU，Context Window = RAM，**Harness = Operating System**，Agent = Application |

---

## 二、范式演进

```
Prompt Engineering  →  Context Engineering  →  Harness Engineering
 "怎么跟模型说话？"     "模型应该看到什么？"      "整个环境应该如何运作？"
```

Harness Engineering 不是替代前两者，而是将其纳入为一个子集。System Prompt、AGENTS.md、渐进式披露都是 Harness 的组成部分。

## 三、六篇文章的核心视角

### 1. Mitchell Hashimoto — 《My AI Adoption Journey》

**贡献：首次提出 "Harness Engineering" 术语并给出操作化定义。**

六步演进框架：
- **放弃聊天机器人** → 使用能读文件、跑命令的 Agent
- **先手动再做一遍** → 把任务拆成小而具体的可验证块
- **下班前启动 Agent** → 深度调研、并行探索、Issue 分类
- **外包确定性任务** → 把 Agent 擅长的活交给后台
- **工程化 Harness** → 每次 Agent 犯错，就加一条规则/工具让它下次不再犯
- **始终有 Agent 在运行** → 目标是 100% 时间有 Agent 后台工作

核心方法论：**规则文件（AGENTS.md）+ 验证工具脚本**，Agent 犯错不是去修 Agent，而是去修 Harness。

---

### 2. Anthropic — 《Effective Harnesses for Long-Running Agents》

**贡献：解决跨会话上下文丢失问题，提出双 Agent 架构。**

核心问题：Agent 每次新会话开始时对之前工作一无所知（"工程师轮班但新来的人没有记忆"）。

两种失败模式：① 一次做太多耗尽上下文；② 过早宣布完成。

解决方案：

| Agent | 职责 |
|-------|------|
| **Initializer Agent** | 首次运行，搭建环境，生成 `init.sh`、`claude-progress.txt`、功能列表 JSON |
| **Coding Agent** | 每次后续会话做增量改动，一次只做一个功能 |

核心机制：功能列表全部初始标记 `failing` → 增量推进 → Git 提交 + 进度文件 → 端到端自验证。

---

### 3. OpenAI — 《Harness Engineering》

**贡献：极端实践验证——0 行人类代码，100 万行 Agent 代码。**

实验数据：3→7 人团队，5 个月，~100 万行代码，~1500 个 PR，人均日 3.5 PR。

六大原则：

| 原则 | 含义 |
|------|------|
| **仓库即唯一事实来源** | Slack/文档/人脑中的知识对 Agent 等同于不存在 |
| **地图而非手册** | AGENTS.md ~100 行，做目录索引，指向深层结构化知识 |
| **机械化强制执行** | 自定义 linter 固化架构不变量，lint 错误内嵌修复指令 |
| **Agent 可读性优先** | 选 API 稳定、训练集覆盖好的"无聊技术"，UI/日志/指标对 Agent 暴露 |
| **高吞吐合并哲学** | 修正成本低，等待成本高，PR 生命周期极短 |
| **熵管理即垃圾回收** | Agent 会复现坏模式，需定期后台清理 Agent 扫描偏差 |

---

### 4. LangChain — 《The Anatomy of an Agent Harness》

**贡献：对 Harness 的组件级解剖，提出 "向后推导" 设计方法。**

从期望行为反向推导 Harness 需要什么：

| 组件 | 功能 |
|------|------|
| **文件系统 + Git** | 最基础的 Harness 原语，持久化工作、跨会话/多 Agent 协作 |
| **Bash + 代码执行** | 通用逃生舱，模型可以自己"发明"工具 |
| **沙箱** | 安全隔离执行，按需创建/销毁，预装运行时和工具链 |
| **记忆与搜索** | AGENTS.md + Web Search + MCP，形成持续学习循环 |
| **上下文腐烂防御** | 压缩、工具输出截断、渐进式披露 |
| **长时自主执行** | Ralph Loop（拦截退出企图，干净上下文窗口重新注入原始 prompt） |
| **自验证循环** | 写代码 → 跑测试 → 检查日志 → 修复 |

关键洞察：同一个模型仅靠更换 Harness，可以从 Terminal Bench 2.0 的 Top 30 跃升到 Top 5。

---

### 5. Anthropic — 《Harness Design for Long-Running Apps》

**贡献：三 Agent 架构（Planner / Generator / Evaluator），解决上下文漂移。**

| 角色 | 职责 |
|------|------|
| **Planner（规划者）** | 将 1-4 句需求扩展为完整产品规格 |
| **Generator（生成者）** | 按 Sprint 逐个实现功能 |
| **Evaluator（评估者）** | 使用 Playwright MCP 独立验收，不共享 Generator 内部状态 |

关键机制：**Sprint Contract**——Generator 和 Evaluator 在每个 Sprint 前协商"完成"的定义；Agent 之间通过结构化工件（JSON/MD）通信。

效果对比：同一条 prompt → 单 Agent 20 分钟/$9 核心功能不可用 → 完整 Harness 6 小时/$200 功能完整可用。

---

### 6. Claude — 《Building Agents with the Claude Agent SDK》

**贡献：Harness 的工程解耦原则。**

三大组件解耦：

```
大脑（Harness） ←→ 手（Sandbox） ←→ 会话（Session Log）
 调用模型、决策       执行代码、操作文件      事件日志、持久化
```

Agent 工作循环：**收集上下文 → 采取行动 → 验证工作 → 重复**

核心设计原则：
- **给 Agent 一台计算机**，不是给一个 API
- **子 Agent 上下文隔离**——只返回摘要，不污染父 Agent
- **容器是"牲畜"不是"宠物"**——挂了随时替换
- **接口优于实现**——像 OS 的 `read()`，只锁定接口，背后实现独立演进

## 四、统一架构模式：三核心循环

综合六篇文章，Harness 的设计可以从三个相互嵌套的核心循环来理解。这三个循环分别回答了"如何组织工作流"、"如何在时间维度上管理知识"、"如何让系统自我进化"。

---

### 4.1 执行循环：Plan → Execute → Verify

Harness 不是单次问答，而是一个持续运行的代理循环（Agent Loop）。这个循环将任务分解为专门的子智能体角色，形成结构化的三阶段流水线：

```
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │  PLANNER  │ ──→ │GENERATOR │ ──→ │EVALUATOR │
  │  规划者    │     │  生成者   │     │  评估者   │
  │ "做什么"   │     │  "怎么做"  │     │ "做对了吗" │
  └──────────┘     └──────────┘     └──────────┘
        ↑                                  │
        └────────── 反馈 ──────────────────┘
```

**Plan（规划）**：由 Planner 将模糊目标扩展为详尽规格和可执行任务列表。这不是一次性的——任务被分解为文档化的多步骤计划，智能体之间通过 **Sprint Contract** 明确每个阶段的"完成"定义。

**Execute（执行）**：由 Generator/Coding Agent 按照**深度优先、增量完成**原则推进。一次只做一个功能，完成后立即 Git 提交并更新进度文件（如 `claude-progress.txt`），确保每次会话结束时代码处于"可直接合入主分支"的干净状态。

**Verify（验证）**：由 Evaluator 独立验收。关键设计原则是 **Evaluator 不共享 Generator 的内部状态**——它使用浏览器自动化（Playwright）像真实用户一样操作应用，依据预设标准独立打分，确保纠偏能力不被生成过程的上下文污染。

---

### 4.2 知识循环：上下文工程与跨会话记忆

Harness 需要解决的核心工程问题之一是：Agent 每次新会话启动时对之前工作完全失忆。这不是模型的缺陷，而是上下文窗口的本质限制。Harness 的应对策略构成了一套知识管理循环：

| 策略 | 机制 | 解决的问题 |
|------|------|------------|
| **结构化交接** | `claude-progress.txt`、Git 提交历史、功能列表 JSON | 上一会话的 Agent 为下一会话的 Agent 留下可读的状态快照 |
| **环境初始化** | `init.sh` 等启动脚本 | 每个新会话从干净、已知的环境快速启动，不浪费 token 摸索运行方式 |
| **地图而非手册** | AGENTS.md ~100 行做目录索引，指向 `docs/` 深层知识 | 渐进式披露——Agent 先读小而稳定的入口，按需深入 |
| **上下文主动重置** | 当上下文窗口接近饱和时，主动截断并启动新会话 | 阻止"上下文腐烂"——模型在过长的上下文中失去一致性 |
| **子智能体上下文隔离** | 每个子 Agent 拥有独立上下文窗口，只向父 Agent 返回摘要 | 噪声不跨 Agent 传播，每个 Agent 的推理空间保持清洁 |
| **文档园丁（Doc Gardener）** | 专门的 Agent 持续扫描文档与代码的偏差，自动发起修正 PR | 知识库不会腐烂，Agent 看到的始终是当前真相 |

这套机制的本质上是在做一件事：**将 Agent 的"记忆"从不可靠的上下文窗口内，迁移到可靠的文件系统和版本控制中。** 文件系统中的内容是持久化事实——可以被读取、验证、修改，不随会话消失。

---

### 4.3 进化循环：Ralph Wiggum 反馈闭环

这是 Harness 工程最核心的方法论，以 Mitchell Hashimoto 的表述最为精炼：

> **"每次 Agent 犯错，花时间工程化一个方案，让它永远不会再犯同样的错误。"**

这个循环被称为 **Ralph Wiggum Loop**（名字来自《辛普森一家》中不断犯错又不断尝试的角色），其运转方式为：

```
Agent 犯错 → 人类分析根因 → 添加规则/工具/lint → Agent 被约束 → 不再犯同样错误
     ↑                                                              │
     └──────────────── 持续迭代 ────────────────────────────────────┘
```

具体手段包括：

| 反馈类型 | 示例 | 约束强度 |
|----------|------|----------|
| **规则反馈** | 自定义 linter、架构不变量检查、类型系统 | 机械强制执行，无法绕过 |
| **视觉反馈** | Playwright 截图对比、UI 回归测试 | 可观测的客观证据 |
| **LLM-as-Judge** | 另一个模型对输出独立评分 | 弱于前两者，但适用范围更广 |
| **人类品味编码** | 将架构原则、编码规范写成可执行脚本 | 隐性知识 → 显性约束 |

值得注意的是，Harness 本身也需要持续迭代。模型能力变强时，一些曾经必要的组件可能变成累赘——需要定期检视并拆除过时的约束。Harness 的复杂度本身也是一种成本，需要在约束效果和维护成本之间权衡。

## 五、Harness 的垂直分层

从另一个维度，Harness 可以按功能垂直切分为三层：

```
┌─────────────────────────────────────────────┐
│              CONTROL HARNESS                 │
│  权限控制 · 环境隔离 · 行为约束 · 速率限制    │
│  "能不能做、做到什么程度"                     │
├─────────────────────────────────────────────┤
│            EVALUATION HARNESS                │
│  自动测试 · 评分 · 验收 · Lint · 自验证循环   │
│  "判断做得对不对"                             │
├─────────────────────────────────────────────┤
│             AGENT HARNESS                    │
│  模型调用 · 工具/Skills/MCP · 任务拆解        │
│  子Agent编排 · 压缩 · 记忆 · 上下文管理       │
│  "做事"                                      │
└─────────────────────────────────────────────┘
```

三层与三个循环的对应关系：Agent Harness 承载执行循环，Evaluation Harness 承载验证环节和反馈闭环，Control Harness 为所有循环提供安全边界。

## 六、第一性原理：Truth Gap（事实缺口）

### 幻觉和漂移的根源不是模型"不够聪明"

模型在做推理时，它的"事实基础"来自两个来源：

| 来源 | 性质 | 问题 |
|------|------|------|
| **训练数据** | 静态、过时、泛化的概率分布 | 不是当前任务的具体事实 |
| **上下文窗口中的历史输出** | 模型自己生成的内容 | 自循环——错误会自我强化 |

两者都不是**可观测的、可验证的、当前任务空间的 ground truth**。幻觉和漂移不是模型的 bug，而是**封闭推理系统在没有外部事实约束下的必然行为**。

### 自我循环 vs. 事实约束

回头看六篇文章，几乎所有关键 Harness 机制都是在做同一件事：**注入外部事实来打断模型的自我循环**。

```
模型自我循环（幻觉/漂移模式）：
  模型输出 → 模型读取自己的输出 → 基于自己的输出再推理 → 漂移加剧

Harness 注入事实（约束模式）：
  模型输出 → 写入文件/执行代码/运行测试 → 读取外部结果（truth）→ 受约束的推理
```

具体印证：

| 机制 | 如何打破自我循环 |
|------|------------------|
| **Evaluator Agent（Anthropic）** | 不共享 Generator 内部状态，用 Playwright 独立验收——拒绝模型自我评估，强制引入外部视角的 truth |
| **机械化 Lint 强制执行（OpenAI）** | Lint 输出是硬事实——代码要么通过要么不通过，模型不能"说服自己通过了" |
| **功能列表全部 `failing`** | 反直觉但极其有效——让模型去**证明**功能通过（需外部测试证据），而不是**声称**功能通过（只需内部推理） |
| **Git 作为 Truth Ledger** | `git diff`/`git log` 输出不可辩驳——什么被修改、何时、由谁。不是推理，是记录 |
| **文件系统作为持久化中间态** | 将模型推理结果物化为可检查的文件，而非留在上下文窗口的"想法"中 |
| **Sprint Contract** | Generator 和 Evaluator 协商的"完成"定义是书面契约，不是模型心中的模糊概念 |

### Harness 设计的第一性原理

> **推理的质量上限 = 可观测事实对推理的约束强度**

如果推理链中每一步都有对应的外部可观测事实作为锚点，幻觉空间就被大幅压缩。反之，推理链越长、中间没有事实注入点，漂移就呈指数级增长。

这与卡尔·波普尔的科学哲学一致：理论（模型推理）不能被证实，只能被**证伪**。好的 Harness 不是让模型"更正确"，而是让模型**更快地发现自己的错误**——通过持续暴露在可观测事实面前。

### 事实注入的强度层级

在 Harness 的关键切面上，注入的内容有不同的约束强度：

| 注入内容 | 性质 | 约束强度 |
|----------|------|----------|
| 另一个模型的输出 | 仍然是推理 | 弱——只是更多推理叠加 |
| Lint/编译器/测试框架的输出 | 不可辩驳的外部事实 | 强——要么通过，要么不通过 |
| `git diff` 的结果 | 不可辩驳的变更记录 | 强——代码实际变化，不是意图 |
| 浏览器截图（Playwright） | 可观测的 UI 事实 | 强——页面实际渲染结果，不是想象 |
| 模型自我评估/反思 | 推理 | 弱——"我检查过了，应该没问题" |

**一个好的 Harness 设计者，本质上是在做一件事：在每个关键推理节点之后，找到并注入该节点对应的最小可验证事实。**

## 七、核心共识

六篇文章 + 深度分析指向同一个结论：

> **真正的瓶颈不是"模型不够聪明"，而是"环境定义不够清晰"——具体而言，是推理链上可观测事实的密度不够。**

Harness Engineering 的本质是将工程严谨性从"写代码"迁移到"设计约束系统"——人类角色从代码实现者转变为系统导演：**Humans steer, agents execute.**

而 "steer" 的核心动作，就是**持续地向推理循环中注入不可辩驳的外部事实**。
